---
title: Legacy Hierarchy Building
---

This document explains how to use the `@Azure.ClientGenerator.Core.Legacy.hierarchyBuilding` decorator to change the base type of a model in generated client SDKs.

:::caution
Legacy hierarchy building is NOT a recommended pattern and should only be used when specifically requested by SDK architects.
:::

## Overview

The `@hierarchyBuilding` decorator changes the base type (parent class) of a model in the generated SDK type graph. It does **not** validate property presence at decoration time. Instead, property reconciliation happens during SDK type graph building.

Use `@hierarchyBuilding` when you need to:

- Rebase a model onto a different ancestor in the inheritance chain (e.g., skip intermediate models)
- Rebase a model onto an unrelated base that shares common properties (e.g., brownfield ARM resources onto `TrackedResource`)
- Create multi-level discriminated hierarchies in generated SDKs

## Property Reconciliation Rules

When the decorator is applied, the following rules govern how properties are reconciled:

### 1. Property Lifting

Properties from removed intermediate ancestors (between the target's original parent and the new base) are "lifted" onto the rebased model as its own properties.

<ClientTabs>

```typespec
model C {
  c?: string;
}
model B extends C {
  b?: string;
}
@Azure.ClientGenerator.Core.Legacy.hierarchyBuilding(C)
model A extends B {
  a?: string;
}
// SDK result: A extends C. A.properties = { a, b }. C.properties = { c }.
// 'b' was lifted from removed intermediate B.
```

```python
class C:
    c: str | None

class A(C):
    a: str | None
    b: str | None
```

```csharp
public partial class C
{
    public string? CValue { get; set; }
}

public partial class A : C
{
    public string? AValue { get; set; }
    public string? B { get; set; }
}
```

```typescript
export interface C {
  c?: string;
}

export interface A extends C {
  a?: string;
  b?: string;
}
```

```java
public final class C {
    private String c;
}

public final class A extends C {
    private String a;
    private String b;
}
```

```go
type C struct {
	C *string
}

type A struct {
	C
	A *string
	B *string
}
```

</ClientTabs>

### 2. Duplicate Dropping

Properties whose names are supplied by the new base chain are dropped from the model (they are inherited instead).

<ClientTabs>

```typespec
model B {
  propB: string;
}
model A {
  ...B;
  propA: string;
}
@@Azure.ClientGenerator.Core.Legacy.hierarchyBuilding(A, B);
// SDK result: A extends B. A.properties = { propA }. B.properties = { propB }.
// propB was A's own property via spread, but is now supplied by new base → dropped.
```

```python
class B:
    prop_b: str

class A(B):
    prop_a: str
```

```csharp
public partial class B
{
    public string PropB { get; }
}

public partial class A : B
{
    public string PropA { get; }
}
```

```typescript
export interface B {
  propB: string;
}

export interface A extends B {
  propA: string;
}
```

```java
public final class B {
    private String propB;
}

public final class A extends B {
    private String propA;
}
```

```go
type B struct {
	PropB string
}

type A struct {
	B
	PropA string
}
```

</ClientTabs>

### 3. Type Compatibility Check

When dropping a property, if the type on the dropped property is incompatible with the same-named property on the new base chain, a `legacy-hierarchy-building-conflict` warning with `property-type-mismatch` message is emitted. The property is still dropped.

<ClientTabs>

```typespec
model C {
  shared?: int32;
}
model OldBase {
  shared?: string; // different type than C.shared
}
@Azure.ClientGenerator.Core.Legacy.hierarchyBuilding(C)
model A extends OldBase {
  a?: string;
}
// SDK result: A extends C. A.properties = { a }.
// Warning emitted: 'shared' type mismatch (string vs int32).
```

```python
# legacy-hierarchy-building-conflict warning is emitted during generation.
class C:
    shared: int | None

class A(C):
    a: str | None
```

```csharp
// Warning: property-type-mismatch for 'shared' on hierarchy rebasing.
public partial class C
{
    public int? Shared { get; set; }
}

public partial class A : C
{
    public string? AValue { get; set; }
}
```

```typescript
// Warning: property-type-mismatch for "shared".
export interface C {
  shared?: number;
}

export interface A extends C {
  a?: string;
}
```

```java
// Warning: property-type-mismatch for "shared".
public final class C {
    private Integer shared;
}

public final class A extends C {
    private String a;
}
```

```go
// Warning: property-type-mismatch for "shared".
type C struct {
	Shared *int32
}

type A struct {
	C
	AValue *string
}
```

</ClientTabs>

Compatible types are silently dropped without warnings:

<ClientTabs>

```typespec
scalar azureLocation extends string;
model C {
  kind: string;
  location: string;
}
model OldBase {
  kind: "old";
  location: azureLocation;
}
@Azure.ClientGenerator.Core.Legacy.hierarchyBuilding(C)
model A extends OldBase {
  a?: string;
}
// SDK result: A extends C. A.properties = { a }.
// No warning: literal "old" is assignable to string, azureLocation is assignable to string.
```

```python
class C:
    kind: str
    location: str

class A(C):
    a: str | None
```

```csharp
public partial class C
{
    public string Kind { get; }
    public string Location { get; }
}

public partial class A : C
{
    public string? AValue { get; set; }
}
```

```typescript
export interface C {
  kind: string;
  location: string;
}

export interface A extends C {
  a?: string;
}
```

```java
public final class C {
    private String kind;
    private String location;
}

public final class A extends C {
    private String a;
}
```

```go
type C struct {
	Kind string
	Location string
}

type A struct {
	C
	AValue *string
}
```

</ClientTabs>

### 4. Discriminator Preservation

Discriminator properties are never dropped, even if the new base has a same-named property.

## Common Use Cases

### Multi-Level Discriminated Hierarchy

Use the decorator to create deeper inheritance in discriminated models:

<ClientTabs>

```typespec
@discriminator("kind")
model Animal {
  kind: string;
  name: string;
}

alias PetContent = {
  trained: boolean;
};

model Pet extends Animal {
  kind: "pet";
  ...PetContent;
}

alias DogContent = {
  breed: string;
};

@Azure.ClientGenerator.Core.Legacy.hierarchyBuilding(Pet)
model Dog extends Animal {
  kind: "dog";
  ...PetContent;
  ...DogContent;
}
// SDK result: Dog extends Pet. Dog.properties = { kind, breed }.
// 'trained' is supplied by Pet → dropped. 'kind' is a discriminator → preserved.
```

```python
class Animal:
    kind: str
    name: str

class Pet(Animal):
    kind: Literal["pet"]
    trained: bool

class Dog(Pet):
    kind: Literal["dog"]
    breed: str
```

```csharp
public partial class Animal
{
    public string Kind { get; }
    public string Name { get; }
}

public partial class Pet : Animal
{
    public bool Trained { get; }
}

public partial class Dog : Pet
{
    public string Kind { get; }
    public string Breed { get; }
}
```

```typescript
export interface Animal {
  kind: string;
  name: string;
}

export interface Pet extends Animal {
  kind: "pet";
  trained: boolean;
}

export interface Dog extends Pet {
  kind: "dog";
  breed: string;
}
```

```java
public class Animal {
    private String kind;
    private String name;
}

public class Pet extends Animal {
    private Boolean trained;
}

public class Dog extends Pet {
    private String kind;
    private String breed;
}
```

```go
type Animal struct {
	Kind string
	Name string
}

type Pet struct {
	Animal
	Trained bool
}

type Dog struct {
	Pet
	Kind string
	Breed string
}
```

</ClientTabs>

### Brownfield ARM Resource Rebasing

A common pattern for Azure resources that need to be rebased onto `TrackedResource`:

<ClientTabs>

```typespec
model Resource {
  id?: string;
  name?: string;
  type?: string;
}
model TrackedResource extends Resource {
  location: string;
  tags?: Record<string>;
}
model Foo extends Resource {
  properties: FooProperties;
  location?: string;
  tags?: Record<string>;
}
@@Azure.ClientGenerator.Core.Legacy.hierarchyBuilding(Foo, TrackedResource);
// SDK result: Foo extends TrackedResource. Foo.properties = { properties }.
// location and tags are supplied by TrackedResource → dropped.
// id, name, type are supplied by Resource (in TrackedResource's chain) → dropped.
```

```python
class TrackedResource(Resource):
    location: str
    tags: dict[str, str] | None

class Foo(TrackedResource):
    properties: FooProperties
```

```csharp
public partial class TrackedResource : Resource
{
    public string Location { get; }
    public IDictionary<string, string>? Tags { get; }
}

public partial class Foo : TrackedResource
{
    public FooProperties Properties { get; }
}
```

```typescript
export interface TrackedResource extends Resource {
  location: string;
  tags?: Record<string, string>;
}

export interface Foo extends TrackedResource {
  properties: FooProperties;
}
```

```java
public class TrackedResource extends Resource {
    private String location;
    private Map<String, String> tags;
}

public class Foo extends TrackedResource {
    private FooProperties properties;
}
```

```go
type TrackedResource struct {
	Resource
	Location string
	Tags map[string]string
}

type Foo struct {
	TrackedResource
	Properties FooProperties
}
```

</ClientTabs>
