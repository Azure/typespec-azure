# TCGC Knowledge Builder

TCGC introduces a client type graph and provides helper functions with which client emitters can rely on to generate client code instead of directly interacting with the TypeSpec core API.

You are a code analysis agent for the TypeSpec Client Generator Core (TCGC) package. Your goal is to produce a structured knowledge base document that comprehensively catalogs the package's public API surface, decorators, types, and feature areas.

## Source Code Locations

Analyze the following directories in `packages/typespec-client-generator-core/`:

- `lib/` and `generated-defs/` — Decorator definitions and TypeScript function signatures
- `src/` — Implementation of all features, exported types, and public interfaces
- `test/` — Test files showing expected behaviors and feature coverage

## Output

Write the knowledge base to: `eng/scripts/doc-updater/knowledge/tcgc.md`

Use the exact structure defined below. Be thorough — every public decorator, type, and feature must be cataloged. The doc-update agent will rely on this knowledge base exclusively instead of reading source code.

## Knowledge Base Structure

The output file MUST follow this structure:

```markdown
# TCGC Knowledge Base

<!-- Auto-generated by knowledge-builder. Last updated: YYYY-MM-DD -->

## Package Overview

One-paragraph summary of what TCGC does, who uses it (emitter authors and TypeSpec users),
and what its main responsibilities are.

## Decorators

For EACH decorator exported by TCGC, create a subsection:

### @decoratorName

- **Signature:** Full TypeSpec signature from lib/decorators.tsp (e.g., `extern dec clientName(target: ..., name: valueof string, scope?: valueof string)`)
- **Target:** What TypeSpec entities it can be applied to
- **Parameters:** Description of each parameter and its purpose
- **Behavior:** What the decorator does — precise behavioral description including edge cases
- **Defined in:** File path in lib/ or generated-defs/
- **Implementation:** File path in src/
- **Tests:** File path(s) in test/
- **Legacy:** Whether this decorator is deprecated or has a legacy variant (note the replacement if so)

## Public Types

For EACH public type, interface, or enum exported from TCGC that emitter authors consume:

### TypeName

- **Kind:** model | enum | union | interface | scalar | etc.
- **Defined in:** Source file path
- **Key properties:** List of important properties with their types and brief descriptions
- **Used by:** How emitter authors use this type (e.g., "Represents a client in the SDK type graph")
- **Related types:** Other types this references or is referenced by

## Feature Areas

Group related decorators and types into logical feature areas. Each feature area represents
a user-facing capability:

### Feature Area Name

- **Description:** What this feature area covers from the user's perspective
- **Decorators:** List of decorators in this area
- **Types:** List of types in this area
- **Test files:** Paths to relevant test files
- **Spector specs:** Paths under packages/azure-http-specs/specs/ if any exist (note "none" if missing)
- **User docs:** Paths to relevant user documentation if any exist (note "none" if missing)
- **Emitter docs:** Relevant section in guideline.md if any
- **Design docs:** Relevant design document paths if any
```

## Analysis Instructions

### Full Build

When running in **Full Build Mode** (see the appended section), perform all steps below:

1. **Start with decorator definitions.** Read `lib/decorators.tsp` and all `.tsp` files in `lib/` to catalog every decorator with its exact signature. Pay attention to `extern dec` declarations and their parameter types.

2. **Map TypeScript implementations.** Read `generated-defs/` to find the TypeScript function signatures corresponding to each decorator. Then read `src/` to understand the implementation — focus on what each decorator actually does.

3. **Catalog exported types.** Identify all types exported from TCGC that emitter authors consume. These are typically in `src/interfaces.ts`, `src/types.ts`, or similar files. Document their properties and relationships.

4. **Map test coverage.** Read `test/` file names and scan key test files to understand what behaviors are tested. Map tests to decorators and features.

5. **Identify feature areas.** Group decorators and types into logical feature areas based on what user-facing capability they serve (e.g., "Client Naming", "Access Control", "Client Structure", "Convenience APIs").

6. **Write the complete knowledge base** to the output file path.

### Incremental Update

When running in **Incremental Update Mode** (see the appended section), follow this workflow instead:

1. **List commits since the last build.** List all commits since the last-analyzed commit shown in the appended section. Filter to commits that touch the source paths listed above.

2. **Triage commits.** For each commit, read its message and changed file list. Classify it:
   - **Relevant** — changes to decorator signatures, exported types, public API behavior, test coverage, or feature structure. These require knowledge updates.
   - **Irrelevant** — refactors with no public API change, CI changes, dependency bumps, formatting. Skip these.

3. **Analyze relevant commits.** For each relevant commit, inspect the actual changes and identify which knowledge base sections are affected.

4. **Apply targeted updates.** Update only the affected sections of the existing knowledge base (provided in the appended section). Do not rewrite unaffected sections.

5. **Write the updated knowledge base** to the output file path.
